# 周报配方 · Report Recipes

Backs Stage 2–3 of weekly-report-alchemy.

## 1. 读者分型

| 读者 | 他真正想知道 | 篇幅 | 词汇 |
|------|--------------|------|------|
| 直属上级 | 进度是否可控、哪里需要他出手 | 半页 | 项目黑话可用 |
| 跨部门 | 与他相关的接口和时间点 | 5 行内 | 去黑话,讲影响 |
| 大老板 | 业务结果与风险,不要过程 | 3–5 行 | 只讲结果与数字 |

同一周可生成多版本;先产直属上级版,其余按它裁剪。

## 2. 提纯改写规则

公式:**动作(具体动词)→ 结果(可验证)→ 影响(对谁、多少)**

| 流水账 | 炼后 |
|--------|------|
| 推进了新版落地页 | 上线落地页 v2,首屏加载 3.1s→1.4s,注册转化 +18% |
| 参与了三场客户会议 | 完成 3 家客户回访,确认 2 家续约意向,1 家风险(见下) |
| 配合测试修了一些 bug | 关闭 P1 缺陷 4 个,本迭代 P1 清零 |

禁用词(出现即改写):赋能、抓手、闭环、持续推进、积极配合、初步取得成效。

## 3. 量化优先级

能拿到的数字按这个顺序找:业务结果(营收/转化/留存)> 交付量(上线/关闭/产出件数)
> 效率(耗时对比)> 覆盖(人数/场次)。只有"开了几次会"这种数字宁可不放。

## 4. 风险与求助的句式

固定三段式:**现状一句 + 影响一句 + 具体的要**。

> 例:供应商接口联调延期 3 天(现状),按当前进度上线日有 2 天风险(影响),
> 需要您协调对方技术负责人本周内入群(要)。

"要"必须具体到人、事、时限。没有"要"的风险条目降级为"进行中"。

## 5. 下周计划的可验证表述

每条计划下周必须能用"做没做、到没到"回答:

- ✗ 继续推进 A 项目 → ✓ A 项目完成需求评审并冻结范围
- ✗ 优化转化率 → ✓ 上线注册页 A/B 实验,周五出首批数据

计划最多 3 条;超过 3 条说明没有优先级,帮用户砍。
